Cleaning and Analysis

Real data is messy. Prices go missing. Categories are mislabelled. Entries are duplicated. This chapter teaches you how to detect and handle these problems — and then brings everything together in a complete food price analysis.

Part Six

      
Combining DataFrames — concat, merge, and join


      
Real projects rarely live in a single table. You might have price data from two different supermarkets in separate files, or product details in one table and prices in another. pandas gives you three main tools for combining DataFrames: pd.concat() stacks them, pd.merge() joins them on a shared column, and .join() joins them on their index.


      
pd.concat() — stack DataFrames


      
pd.concat() glues DataFrames together end-to-end. By default it stacks rows (adding more observations). Pass axis=1 to stack columns side by side instead.

Part Eight

      
Wide Data vs Long Data — and How to Convert Between Them


      
Imagine you are tracking the price of five grocery items across three months. There are two completely different ways to arrange that information in a table, and the choice has enormous consequences for what you can do with the data afterwards.


      
What is wide data?


      
In a wide table, each subject (here, each grocery item) occupies exactly one row, and every measurement gets its own column. The table literally grows wider as you add more measurements.


      

        
Python · See it — a wide price table

        
        

          
          
          
        

        

      


      
Wide format is natural for humans. A journalist glancing at that table immediately sees that Olive Oil jumped from €5.49 to €5.99. It is the format that Excel spreadsheets, government statistical publications, and most printed reports use. It is compact — five items across three months fit in five rows and four columns.


      
What is long data?


      
In a long table, each individual measurement gets its own row. Instead of spreading months across columns, you add a variable column (which month?) and a value column (what was the price?). The table grows taller as you add more measurements — five items across three months produce fifteen rows.


      

        
Python · See it — the same data in long format

        
        

          
          
          
        

        

      


      
Why would anyone use wide format?


      
Wide format wins whenever a human needs to read the table directly:

      
    
        
  • Side-by-side comparison. It is much easier to scan a row and compare January vs February vs March than to hunt through fifteen rows.
    • 
        
  • Compact reporting. A published table with five rows fits on one page. A fifteen-row table does not.
    • 
        
  • Spreadsheet formulas. Excel and Google Sheets functions like SUM(B2:D2) assume your data is wide.
    • 
        
  • Certain statistical software. Some older tools (SPSS, certain R functions) expect one row per subject.
    • 
        
  • Data entry. It is faster and less error-prone for a human to fill in one row per item than to create a new row for every month.
    • 
      


      
Why would anyone use long format?


      
Long format wins whenever a computer needs to process the data:

      
    
        
  • groupby and aggregation. df.groupby("month")["price"].mean() works instantly on long data. On wide data, you would have to select columns by name and write separate logic for each month — and adding a fourth month means rewriting your code.
    • 
        
  • Filtering. df[df["month"] == "feb"] is one line. Filtering wide data for a specific month means knowing and referencing an exact column name.
    • 
        
  • Plotting. Seaborn, Plotly, and most visualisation libraries expect long format. They need a single column to map to the x-axis, a single column for y, and a single column for colour — wide data cannot provide that cleanly.
    • 
        
  • Scalability. If you add a fourth month to long data, you add five rows. If you add a fourth month to wide data, you add a whole new column — which breaks every formula, query, and chart that referenced column positions.
    • 
        
  • Databases and SQL. Relational databases are built around long format. A SQL query like GROUP BY month requires a column called month.
    • 
      


      

        The rule of thumb: Collect and store data in long format. Reshape it to wide format only at the last step, when you need to present it to a human reader. This is the same principle as keeping a database normalised and only denormalising for a report.

It creates two new columns: one holding the old column names and one holding the old cell values.

Part Nine

      
Applying Your Own Functions to a DataFrame


      
Pandas built-in methods cover a lot of ground, but sooner or later you will need logic that does not exist as a ready-made function. The .apply() method bridges that gap: it calls any Python function you write — once per cell, once per row, or once per column — and collects the results back into the DataFrame.

Scraped product names are rarely clean. The same item might arrive as "  ελαιόλαδο  ", "ΕΛΑΙΌΛΑΔΟ", or "Elaiolado (extra virgin)" depending on the source. Before you can group, merge, or compare, you need every name reduced to a single canonical form. This is exactly the kind of multi-step logic that belongs in a def.

Lambda vs def — when to use which


      
A lambda is an anonymous function — a function with no name, written in a single expression instead of a full def block. The syntax is:


      
lambda argument : expression


      
The word lambda replaces def. Everything to the left of the colon is the parameter list. Everything to the right is the return value — no return keyword needed, because a lambda always returns the result of that one expression.


      
So lambda p: round(p * 1.24, 2) is exactly the same as writing:


      
def apply_vat(p):
    return round(p * 1.24, 2)


      
The difference is that a lambda is created inline, at the point where it is used, and then discarded. It has no name you can call elsewhere in your code, and it cannot contain more than one expression — no if/elif chains, no loops, no multiple statements. That limitation is also its strength: when you read .apply(lambda p: round(p * 1.24, 2)), the entire logic is visible right there in the line, without having to scroll up to find the function definition.


      
For very short transformations — one expression, no guard clause — a lambda avoids the overhead of naming a function. For anything with more than one step, a named def is clearer and easier to test.


      

        
Python · Lambda for simple transforms, def for complex ones

        
        

          
          
          
        

        

      


      
Series.map() — functions, dictionaries, and lookup tables


      
.map() works element-wise on a single Series, exactly like Series.apply() when you pass it a function. The critical difference is that .map() also accepts a dictionary or another Series as its argument, turning it into a vectorised lookup table — something .apply() cannot do.

DataFrame.map() — apply a function to every cell


      
DataFrame.map() (added in pandas 2.1, replacing the older applymap()) calls a function on every individual cell in the entire DataFrame. Use it when you want to transform all values in one pass — for example, rounding every number in a table of prices.


      

        
Python · DataFrame.map() — round every numeric cell

        
        

          
          
          
        

        

      


      

        Choosing between apply, map, and map(dict):
        Use Series.apply(func) or Series.map(func) interchangeably for element-wise work on one column — they produce the same result. Reach for Series.map(dict) whenever you need a lookup table; it is faster and more readable than an equivalent if/elif chain inside .apply(). Use DataFrame.apply(axis=1) when your function needs values from more than one column. Use DataFrame.map(func) when you want to transform every cell in the table at once.
      


      

        A note on speed: .apply() calls your Python function in a loop, one value at a time. For very large datasets this can be slow. If a vectorised pandas method exists — .str.lower(), .str.strip(), arithmetic operators — prefer it over .apply(). Use .apply() when you need logic that pandas cannot express natively: multi-step conditionals, external lookups, or anything that requires a full Python function.
