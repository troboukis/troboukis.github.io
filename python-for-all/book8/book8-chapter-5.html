Chapter Five

Cleaning Your Data

Raw data is almost never analysis-ready. Cleaning is systematic, documented, and reversible — not a series of manual edits that no one can later verify.

Part One

Diagnosing the Problems

A structured cleaning process begins with diagnosis. Before fixing anything, catalogue what is wrong. Go through every column and record the issues you find.

  • Inconsistent formatting: dates in multiple formats, amounts with and without currency symbols
  • Encoding problems: garbled characters, mixed scripts, smart quotes where straight quotes are expected
  • Inconsistent naming: the same institution appearing as three slightly different strings
  • Mixed types in a single column: numbers stored as text, floats where integers are required
  • Implausible values: negative ages, amounts of zero where zero is not a valid entry, future dates in historical records

Write this list down. Every item becomes a cleaning rule — a decision you will apply consistently across the entire dataset and record in your cleaning log.

Part Two

Normalising Key Fields

Normalisation means converting inconsistent representations into a single standard form. The fields that most commonly require it:

Field type Common problem Approach
Amounts Mixed currency symbols, commas as thousand separators Strip non-numeric characters, convert to float
Dates Multiple formats in the same column Parse to a single ISO standard (YYYY-MM-DD)
Names Abbreviations, spelling variants, transliterations Fuzzy matching against a canonical reference list
Identifiers Leading zeros dropped, mixed formats Pad and reformat to the official standard
Municipalities Historical name changes, spelling variants Join to an official administrative reference list

The join keys — the fields used to merge two datasets — deserve particular care. A join on a name field will fail silently if one dataset spells a municipality differently from another. Always verify joins by checking the count of matched and unmatched records.

Part Three

Duplicates, Conflicts, and Outliers

Three categories of data quality problem require individual investigation rather than bulk rules.

Duplicates are rows that appear more than once when they should appear once. A duplicate may be exact — identical values in every field — or near-duplicate: the same entity recorded twice with a slight difference in one field. Investigate each type separately. An exact duplicate is almost always an error; a near-duplicate may represent two genuinely different events.

Conflicting records describe the same entity with different values for the same field. Two rows that agree on an institution's name but disagree on its budget require a decision: which source is authoritative? Document the decision and the reason.

Outliers are values that are statistically extreme relative to the rest of the dataset. An outlier is not automatically an error — it may be the most important data point in the entire investigation. Treat each one as a question, not an answer. Investigate before excluding.

Part Four

Clean, Reproducible Code

Every cleaning step must be written as code — not applied by hand in a spreadsheet. If you discover an error in the raw data after publication, you need to be able to rerun the entire cleaning pipeline from scratch and produce a corrected dataset quickly. Manual edits make this impossible.

Keep a cleaning log alongside the code: a record of every decision you made, including cases where you excluded rows and the reason for the exclusion. The log should read as a clear account of your choices, not just a list of operations.

Never modify the raw data. Save the original file exactly as received, in a folder that your scripts read from but never write to. Every transformation happens in code, on a copy. This means you can always return to the starting point.
